## Zapier-managed path — not usable for Clay

Running any tool through the Zapier connection (`--connection zapier:<id>`)
fails at the relay:

```
ZapierRelayError: Clay does not support direct HTTP requests.
Use zapier.apps.{appKey} or zapier.runAction() to use Actions built for this app.
```

Clay's Zapier relay only exposes pre-built Actions, not the direct-HTTP
passthrough this connector issues. Direct mode (`env:CLAY_API_KEY`) is therefore
the only working path for v1 — consistent with the connector's design and the
MR's "Zapier-managed path unverified" caveat.

## Bugs found and fixed

### `listWorkspaceUsers` — `users[].id` type mismatch ✅ fixed

```
Output validation failed for "listWorkspaceUsers":
✖ Invalid input: expected string, received number  → at users[0].id
```

Clay returns the user id as a **number**; the output schema typed it as
`string`. Fixed by coercing `users[].id` to a string (`z.coerce.string()`),
consistent with `getCurrentUser`'s string `userId`. Re-verified live: returns
`id` as a string, no validation error.

### `updateRecord` — output shape mismatch ✅ fixed

```
Output validation failed for "updateRecord":
✖ Invalid input: expected string, received undefined  → at id
```

Clay's update endpoint returns `{ "message": "Record updates enqueued" }`, not
the updated record. The output schema expected `{ id, cells }` (copied from
`createRecord`). Fixed by modelling the ack (`{ recordId, message }` — echoing
the input `recordId`) and documenting the async ("enqueued") semantics. The
write itself always succeeded — verified by re-reading the row with `findRecord`.
Re-verified live: returns `{ recordId, message }`, no validation error.
